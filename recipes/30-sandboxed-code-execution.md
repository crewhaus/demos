# Recipe 30 — Sandboxed Code Execution

Give the agent `Python`, `JavaScript`, and `Shell` REPL tools that
run inside Docker sandboxes — network=none, read-only root, scratch
tmpfs, image allow-list. Plus seven polyglot images on the same
registry pattern (Go, Rust, Java, Ruby, R, .NET, PHP) so you can
extend the set without changing runtime-core.

You'd reach for sandboxes when:

- The agent should be able to **execute code** as part of solving a
  task (data analysis, calculations, format conversions).
- You're running **untrusted code** generated by the model — never
  unsandbox eval()-style code.
- You're running a **multi-tenant** product and need to bound what
  one tenant's code can do.

For single-host trusted CLIs, plain `Bash` is fine. Sandboxes are
the right answer when the runtime needs **isolation guarantees**.

## Prerequisites

- Docker (or Podman) installed and accessible.
- [Recipe 29 — Permissions Deep Dive](29-permissions-deep-dive.md)
  for tool flag behavior.

## The `sandbox` package

[`packages/sandbox`](https://github.com/crewhaus/factory/blob/main/packages/sandbox) exposes:

```typescript
const sb = createSandbox({
  image: "python:3.13-slim",
  cpu: 1.0,
  memoryMb: 512,
  timeoutMs: 60_000,
  network: "none",
  readonlyRoot: true,
  tmpfs: ["/tmp"]
});

const result = await sb.exec("python3 -c 'print(2+2)'");
// result = { stdout: "4\n", stderr: "", exitCode: 0, durationMs: 142 }
```

Backends:

| Backend  | When to use                                           |
| -------- | ----------------------------------------------------- |
| `docker` | Default. Production.                                  |
| `podman` | Rootless preference (no Docker daemon).                |
| `noop`   | Tests only — opt-in via `CREWHAUS_SANDBOX=noop`. Bypasses sandboxing; in-process. |

## Hardening defaults

The `createSandbox` defaults are the **safe** posture; you have to
opt **out** to relax any of them:

| Constraint        | Default                                      | Why                                              |
| ----------------- | -------------------------------------------- | ------------------------------------------------ |
| Network           | `none`                                       | No outbound; no DNS; no metadata-service access. |
| Memory            | 512 MB                                       | Bound fork-bomb impact.                          |
| CPU               | 1.0                                          | Bound runaway-loop CPU impact.                   |
| Timeout           | 60 s                                         | Kill long-runners.                                |
| Root filesystem   | Read-only                                    | Can't pollute the host or persist payload.       |
| `/tmp`            | tmpfs, 64 MB                                  | Writable scratch; vanishes at container stop.     |
| Allowed mounts    | None (whitelist)                              | No host filesystem leakage.                      |
| Allowed images    | `CREWHAUS_SANDBOX_ALLOWED_IMAGES` env var     | Only vetted images can run.                       |
| User              | UID 1100 (`crewhaus`)                         | Not root inside the container.                   |
| Caps              | All dropped                                   | No `CAP_NET_RAW`, `CAP_SYS_ADMIN`, ...           |

To allow network from one specific sandbox:

```typescript
createSandbox({ image: "python:3.13-slim", network: "bridge" });
```

But this is a **deliberate** opt-out — the tool author has to choose
it. The default is fail-closed.

## The image allow-list

`CREWHAUS_SANDBOX_ALLOWED_IMAGES=python:3.13-slim,node:22-alpine,alpine:3.19`

A sandbox call with an image not in this list fails at construction.
This prevents the agent (or a buggy tool) from pulling an arbitrary
image — which could be malicious or expose untrusted code paths.

Production deployments pin to a small allow-list of CI-built, scanned
images. For the bundled tools the list defaults to the three images
they need.

## Built-in code-execution tools

[`packages/tool-code-execution`](https://github.com/crewhaus/factory/blob/main/packages/tool-code-execution)
ships three tools:

| Tool         | Image                  | Tool flags                                               |
| ------------ | ---------------------- | -------------------------------------------------------- |
| `Python`     | `python:3.13-slim`     | `concurrencySafe: false`, `readOnly: false`, `destructive: true`, `requiresSandbox: true` |
| `JavaScript` | `node:22-alpine`       | Same flag set as Python                                  |
| `Shell`      | `alpine:3.19`          | Same flag set                                            |

All three are `requiresSandbox: true`, which means the permission
engine **denies** them by default if no sandbox is configured. This
is the fail-closed default: an agent can't run these tools unless
the deployment opted into sandboxing.

### Calling

```yaml
agent:
  tools:
    - python      # the model-facing name is Python
    - javascript
    - shell
permissions:
  rules:
    - type: alwaysAllow
      pattern: Python
    - type: alwaysAllow
      pattern: JavaScript
    - type: alwaysAllow
      pattern: Shell
```

The model invokes `Python({ code: "..." })`. The tool spawns the
sandbox, writes the code to `/tmp/script.py`, runs `python3
/tmp/script.py`, and returns `{ stdout, stderr, exitCode }`.

Persistent state across calls within one turn: a tool-level
`workdir` is mounted across calls in the same turn. The model sees
"the same /workdir from the last Python call is still there",
allowing multi-step workflows (write file → process it → analyze
output).

## Warm pool

Each sandbox call has a cold-start cost (~200ms-1s in Docker). For
fast iteration, the runtime maintains a **warm pool**: pre-warmed
containers that wait for the next call.

```bash
CREWHAUS_SANDBOX_WARM_POOL=2 bun run run hello-cli
```

Keeps 2 warm Python containers, 2 warm Node containers, etc. The
first `Python` call lands on a warm container in <50ms. The pool
re-warms after each use.

The pool is **per-image**, **per-deployment**. Multi-tenant managed
deployments scope warm containers per-tenant via
AsyncLocalStorage — tenant-A's warm container is never reused for
tenant-B.

## Streaming output

For long-running calls, stdout/stderr stream through the trace bus
as `tool_stream_chunk` events:

```json
{ "kind": "tool_stream_chunk", "tool": "Python", "stream": "stdout", "data": "step 1 done\n" }
{ "kind": "tool_stream_chunk", "tool": "Python", "stream": "stdout", "data": "step 2 done\n" }
{ "kind": "tool_stream_chunk", "tool": "Python", "stream": "stderr", "data": "WARNING: ...\n" }
```

These flow to OTel exporters and to `CREWHAUS_TRACE=pretty` output,
so you can watch a long-running script's progress without waiting
for completion.

The model **does not** see streaming chunks — only the aggregated
final `tool_result`. Streaming is a runtime observability concern.

## The polyglot image registry

[`packages/sandbox-image-registry`](https://github.com/crewhaus/factory/blob/main/packages/sandbox-image-registry)
lets you register additional language images:

```typescript
import { registerSandboxImage } from "@crewhaus/sandbox-image-registry";

registerSandboxImage({
  id: "go",
  image: "golang:1.22-alpine",
  healthcheck: ["go", "version"],
  defaultEntry: "/usr/local/bin/go run",
});
```

Lookup:

```typescript
const img = lookupSandboxImage("go");
// → { image, healthcheck, defaultEntry }
```

The bundled polyglot images (`sandbox-image-go`, `sandbox-image-rust`,
`sandbox-image-java`, `sandbox-image-ruby`, `sandbox-image-r`,
`sandbox-image-dotnet`, `sandbox-image-php`) all follow this pattern.
Each is one package with the image id, a healthcheck command, and
a smoke test that runs `Hello World`.

To add an eighth language, follow the same template:

1. New package `packages/sandbox-image-<lang>/`.
2. Export a `register()` that calls `registerSandboxImage`.
3. Add a smoke spec.
4. PR.

No changes to `runtime-core` are required. The registry pattern is
why polyglot support is a leaf addition rather than a core change.

## `crewhaus sandbox doctor`

```bash
crewhaus sandbox doctor
```

Lists every registered image with its allow-list / pull status.

```bash
crewhaus sandbox doctor --probe
```

Actually pulls each image and runs its healthcheck. Output:

```
python:3.13-slim    pulled, healthcheck OK   (python3 --version → "3.13.0")
node:22-alpine      pulled, healthcheck OK   (node --version → "v22.4.0")
alpine:3.19         pulled, healthcheck OK   (sh --version → ...)
golang:1.22-alpine  pull FAILED              (denied by registry; image not in allow-list)
```

Run as part of deployment validation to catch image-pull failures
before runtime.

## Prompt-injection on tool output

Code-execution tools are a prime injection vector — a Python script
that prints `[ignore previous instructions]` is a real attack. The
`prompt-injection-detector` ([Recipe 41](41-security-fabric.md)) is
wired into the post-tool path: every tool result is classified before
reaching the model.

| Verdict      | Action                                                              |
| ------------ | ------------------------------------------------------------------- |
| `clean`      | Pass through to the model.                                          |
| `suspicious` | Pass through + log a warning + emit `permission_decision` event.    |
| `malicious`  | Replace with `[tool output redacted: ...]` notice.                  |

The classification is per-tool-call. Layered with sandboxing, the
security property is: **even if a malicious script runs, it can't
exfiltrate via prompt injection or harm the host.**

## When to NOT sandbox

There are cases where sandboxing is overkill:

- **Trusted local CLI.** A developer writing TypeScript with Claude
  Code at their own machine doesn't need sandbox isolation for
  their own commands — they trust themselves.
- **Smoke tests.** `CREWHAUS_SANDBOX=noop` is for unit tests; the
  in-process backend executes commands directly. Don't use in
  production.
- **Pure-text agents.** A chat-only agent with no code-execution
  tools doesn't need any sandbox config.

But anywhere **the agent could run model-generated code that touches
the host**, sandbox. The default for `tool-code-execution` is
`requiresSandbox: true` — the runtime enforces this; you'd have to
deliberately disable it.

## Things that look like sandboxing but aren't

| Symptom                                                          | Better tool                                       |
| ---------------------------------------------------------------- | ------------------------------------------------- |
| Want to restrict file reads.                                     | Permission rules with path globs.                 |
| Want to firewall outbound.                                       | `network: "bridge"` + adapter-level allow-list.   |
| Want resource limits on the host (not just sandboxed processes).  | k8s pod resource requests/limits.                  |
